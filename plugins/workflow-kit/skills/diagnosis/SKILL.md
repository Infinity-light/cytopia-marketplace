## 铁律

不做没有调查的修复。不猜。

## 常见借口（都是错的）

| 你可能会想 | 为什么是错的 |
|-----------|------------|
| "我大概知道原因，直接改了" | 大概 = 猜。猜错了浪费时间，猜对了下次还会猜 |
| "先试试这个修复，不行再查" | 盲目修复会引入新问题，让根因更难定位 |
| "已经修了3次了，再试一次" | 3次失败说明你的假设有问题，不是运气不好 |

# 诊断阶段

## 识别信号

- 用户说"报错了"、"不工作"、"出问题了"
- 执行结果与预期不符
- 用户对某个现象感到困惑
- **TaskCompleted Hook 返回测试失败**（自动触发，无需用户介入）

## 核心流程

### 路径A：用户报告问题（手动触发）

#### 1. 收集信息
了解问题的：
- 具体表现（错误信息、异常行为）
- 发生时机（什么时候开始的）
- 复现条件（如何触发）

#### 2. 探索上下文
查看相关的：
- 代码文件
- 日志输出
- 最近的改动

#### 3. 分析原因
- 形成假设
- 验证假设
- 定位根因

#### 4. 提出解决方案
- 给出修复建议
- 解释为什么会出现这个问题
- 如何避免再次发生

### 路径B：测试失败（Hook 自动触发）

当 TaskCompleted Hook 返回测试失败时，跳过"收集信息"步骤，直接从 Hook 输出中获取：

#### 1. 读取 Hook 输出
- 哪些测试失败了
- 具体错误信息和堆栈
- 对应的测试文件路径

#### 2. 读取相关代码
- 失败的测试文件
- 被测试的实现代码

#### 3. 判断失败类型
- **代码缺陷**：实现不符合交付标准 → 修复代码，返回 execution
- **测试缺陷**：测试本身写错了 → 修复测试，重新运行
- **设计问题**：交付标准或架构有矛盾 → 返回 planning

## 输出

完成诊断阶段后，输出：
- 问题的根本原因
- 解决方案
- 预防建议（可选）

## 何时读取references

- 需要调试策略 → 读取 `references/debugging-strategies.md`
- 需要错误分析方法 → 读取 `references/error-analysis.md`

## 阶段转换

问题解决后：
- **代码缺陷** → 返回 execution，修复后重新标记任务完成
- **测试缺陷** → 修复测试，重新运行验证
- **设计问题** → 返回 planning，修订方案
- **需求变化** → 返回 discovery

## 调试铁律（融入）

### 诊断无副作用
- 诊断阶段只做只读检查，禁止修改状态
- 先观察，后动手

### Console优先
- 先看报错信息，再动手修改
- 错误信息是最直接的线索

### 文档优先
- 官方文档优先于试错
- 遇到不确定的API，先查文档

### 最简验证
- 先用最少参数确认能工作
- 再逐步添加复杂配置

### 总开关影响
- 修改全局配置前，确认所有依赖方

## 调试升级机制

- 第 1-3 次修复尝试：正常调试流程
- **第 3 次失败后**：强制停下
  - 这不是"假设失败"，这是"架构错误"
  - 回退到 Planning 重新审视方案
  - 用 AskUserQuestion 告知用户："连续 3 次修复失败，建议回退到 Planning 重新设计。"
  - 选项：进入 Planning / 继续尝试（需说明新思路） / 暂停

## 阶段结束

问题解决后，用 AskUserQuestion 向用户确认下一步：

选项必须包含：
- "返回 Execution 继续"（推荐）
- "进入 Planning 重新设计"
- "暂停"
