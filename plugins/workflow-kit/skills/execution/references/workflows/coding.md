# Coding Workflow

## 1. 执行前：上下文探索

1. **读取文件契约**
   - 读取目标文件头部的 `---` 契约块，理解文件职责、依赖关系、需要实现的函数
   - 读取对应的测试文件骨架，理解测试用例

2. **读取项目上下文**
   - 如果项目有 CLAUDE.md，优先读取——其中的架构约定和技术规范是最高优先级约束
   - 扫描 `practices/` 目录，找到与当前技术栈匹配的规范文件并读取

3. **Skill 按需加载**
   - 涉及前端 UI 开发 → 加载 `frontend-design` Skill
   - 涉及安全敏感操作 → 读取 `practices/security-checklist.md`

## 2. TDD 串行循环（契约驱动）

前提：Planning 阶段已创建文件骨架（契约）和测试骨架（描述 + TODO）。

```
┌─────────────────────────────────────────────┐
│  Subagent-A: RED（实现测试断言）              │
│  - 读取目标文件头部契约（测试依据）            │
│  - 读取测试骨架文件                           │
│  - 将 TODO 替换为具体断言                     │
│  - 运行测试 → 确认失败（RED）                 │
│  - 产出：失败的测试文件                       │
├─────────────────────────────────────────────┤
│  Subagent-B: GREEN（实现代码）                │
│  - 读取目标文件头部契约（实现规格）            │
│  - 读取 CLAUDE.md（全局上下文）               │
│  - 读取 practices/ 中匹配的规范               │
│  - 写实现 → 运行测试 → 确认通过（GREEN）      │
│  - 失败则自行修复，连续失败 2 次 → 报告主Agent │
│  - 通过后将契约 status 改为 IMPLEMENTED       │
│  - 产出：让测试通过的实现代码                  │
├─────────────────────────────────────────────┤
│  Subagent-C: REFACTOR（可选）                 │
│  - 读取 practices/coding-standards.md         │
│  - 重构 → 运行测试 → 确认仍通过              │
└─────────────────────────────────────────────┘

主Agent 只负责调度和判断流程走向，不直接执行任何具体操作。
连续失败 2 次由 Subagent 报告后，主Agent 决定是否转 diagnosis 阶段。
```

对于简单修改或非核心逻辑，可跳过 RED 直接实现，但必须在实现后运行测试验证。

## 3. 验证流程

按顺序执行，任一阶段失败则停止修复后重来：

1. 构建验证
2. 类型检查
3. Lint 检查
4. 测试套件 — 覆盖率标准参照 `practices/testing-standards.md`
5. 安全扫描 — 检查项参照 `practices/security-checklist.md`
6. Diff 审查 — 检查是否有意外变更、遗漏的边界情况
7. 代码审查 — 启动审查子代理，对照文件头部契约逐条检查实现是否符合规格

E2E 验证在关键功能变更时追加。

## 4. Subagent 调度：约束传递

调度 Subagent 时，**必须在 Task prompt 中显式指定需要读取的文件**：

模板：
```
执行 [具体任务描述]。

开始前先读取以下文件：
1. [目标文件路径]（头部 --- 契约块是你的规格）
2. [测试文件路径]（如适用）
3. 项目 CLAUDE.md（全局上下文和开发规范）
4. practices/[匹配的规范文件]

规则：
- 不要修改 --- 契约块的内容（除 status 字段外）
- 遵循 CLAUDE.md 中的开发规范
```

不要假设 Subagent 会自动继承主 Agent 的上下文——文件路径必须显式传递。
