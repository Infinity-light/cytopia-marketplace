## 核心方法论：三部曲

Discovery 严格遵循"为什么 → 是什么 → 怎么办"的三部曲逻辑：

| 阶段 | 核心问题 | 产出 |
|------|---------|------|
| **为什么** | 用户到底想要什么？ | 清晰的需求边界、交付物清单、验收标准 |
| **是什么** | 这个东西本质上怎么回事？| **最小模块拆解** + **三重维度尽调** |
| **怎么办** | 具体如何组织实现？ | **用户动线梳理** + **细粒度步骤** |

---

## 第一部曲：为什么（Why）

**目标**：透彻探明用户需求，具体到交付物的每一个页面和最小呈现节点。

### 动作清单

1. **接收用户表达**
   - 不打断、不急于给方案
   - 完整听完用户说什么

2. **概念澄清**
   - 对每个关键概念、术语、引用逐一查证
   - 不确定 → 先查（读文件、搜索、问用户）
   - 不假设、不替代、不猜测

3. **需求拆解**
   - 目标：用户想达成什么？
   - 动机：为什么想做这个？
   - 约束：有什么限制条件？
   - 边界：什么**不在**范围内？

4. **交付物具象化**
   - 如果是产品/系统：画出每个页面、每个模块
   - 如果是功能：描述每个交互节点的输入输出
   - 如果是服务：明确调用方和使用场景
   - **拆解到最小可验证单元**（单个子代理可独立完成）

### 检查点

通过以下检查才能进入第二部曲：

- [ ] 用户目标已清晰
- [ ] 约束条件已确认
- [ ] 边界范围已划定
- [ ] 交付物已具象到**最小节点**
- [ ] 没有"我不确定用户想要…"的残留疑问

## 第二部曲：是什么（What）

**核心重点**：
1. **最小模块拆解** - 将需求拆分到不可再分的原子单元
2. **三重维度尽调** - 对每个最小模块进行深度、广度、前瞻性的调研
3. **最前沿方案** - 找到当前技术领域最先进的实现方式

### 第一步：最小模块拆解

将需求拆解为**树状结构**，直到**不可再分的最小功能单元**：

```
用户需求
├── 模块 A
│   ├── A1: 最小功能单元（可被单个命令/函数完成）
│   ├── A2: 最小功能单元
│   └── A3: 最小功能单元
├── 模块 B
│   ├── B1: 最小功能单元
│   └── B2: 最小功能单元
└── 模块 C
    ├── C1: 最小功能单元
    └── C2: 最小功能单元
```

**拆解标准**：
- 每个最小单元可被**单个子代理独立完成**
- 每个最小单元有**明确的输入输出**
- 每个最小单元对应**具体的技术方案选型**

### 第二步：三重维度尽调

对每个**最小模块**，并行发起子代理进行三重尽调：

| 维度 | 调研内容 | 产出要求 | 工具 |
|------|---------|---------|------|
| **深度** | 技术原理、架构设计、实现细节、性能特征 | 原理性解释 + 代码示例 | context7 + 官方文档 |
| **广度** | 所有主流方案对比（开源/商业/云原生）、适用场景、成本分析 | 对比表格（≥3种方案）| WebSearch + GitHub |
| **前瞻性** | 技术演进趋势、新兴方案、即将成为主流的技术 | 趋势判断 + 推荐理由 | 论文搜索 + 社区讨论 |

**尽调深度要求**：
- 开源项目：GitHub stars >1k，查看最近 commit 活跃度
- 技术方案：每个方案列出优缺点、适用场景、学习成本、维护成本
- 坑点分析：从 GitHub issues、Stack Overflow、开发者博客收集已知问题
- 前沿追踪：搜索 2024-2026 年的新技术、新论文、新工具

**产出**：每个最小模块的尽调报告，包含：
- 技术原理解析（深度）
- 全方案对比表（广度）
- 趋势判断与推荐（前瞻性）

## 第三部曲：怎么办（How）

**核心重点**：
1. **用户动线梳理** - 以用户视角，从起点到终点的完整流程
2. **细粒度步骤** - 每一步都具体到可执行级别
3. **逻辑结构组织** - 清晰地展示整体架构和数据流

### 第一步：梳理用户动线

**以用户视角为出发点**，梳理实现该需求的完整流程：

```
用户起点（输入/触发）
    ↓
步骤 1: [具体操作]
    ↓
步骤 2: [具体操作]
    ↓
步骤 3: [具体操作]
    ↓
用户终点（输出/结果）
```

**动线梳理要求**：
- 从用户的**第一个动作**开始（如"输入 URL"、"上传文件"）
- 每个步骤具体到**可执行级别**（不要抽象描述）
- 展示**决策分支**（如"如果 X 则 A，否则 B"）
- 明确**每一步的输出**作为下一步的输入

### 第二步：逻辑结构设计

**以系统逻辑为出发点**，展示内部架构：

```
┌─────────────────────────────────────────┐
│              系统架构图                  │
├─────────────────────────────────────────┤
│                                         │
│  输入层 → 处理层 → 存储层 → 输出层       │
│     ↓        ↓        ↓        ↓        │
│   [细节]   [细节]   [细节]   [细节]     │
│                                         │
└─────────────────────────────────────────┘
```

**架构展示要求**：
- 数据流清晰（箭头指向明确）
- 模块边界清晰（每个模块职责单一）
- 接口契约明确（输入/输出格式）
- 异常处理可见（错误如何传播）

### 第三步：呈现给用户

**让用户知道最细粒度的每一步应该怎么做**：

1. **展示调研结果**
   - 每个最小模块的三重尽调结论
   - 用表格对比不同方案的取舍
   - 给出推荐方案及明确理由

2. **展示动线与架构**
   - 用户动线流程图
   - 系统架构图
   - 模块间的数据流转

3. **用户确认**
   - 使用 AskUserQuestion 呈现选项
   - 每个选项说明各自的取舍
   - 等待用户明确选择（不是"都行"）

### 检查点

- [ ] 每个最小模块的三重尽调已完成
- [ ] 用户动线已梳理（从起点到终点）
- [ ] 系统架构已设计（模块+数据流）
- [ ] 用户已明确选择（不是"都行"）

## 输出：PRD 文档

### 命名规范

PRD 文件必须按以下格式命名，避免重复和混淆：

```
[项目简短描述] PRD.md
```

**命名要求**：
- 包含项目核心关键词（如功能名、系统名）
- 如果是 Skill，标注 "Skill"
- 示例：
  - `视频语音处理 Skill PRD.md`
  - `Claude 安装器打包 PRD.md`
  - `舆情监测系统 PRD.md`

**存储位置**：`.claude/discovery/[命名].md`

## 阶段结束

**【强制规则】Discovery 完成后必须进入 Planning，不允许跳过。**

用 AskUserQuestion 向用户确认：

选项必须包含：
- "进入 Planning"（唯一选项）
- "需要补充调研"（回到第二部曲）

**不允许的选项：**
- ~~"跳过 Planning 直接进入 Execution"~~ ❌
- ~~"需求已经很清楚了，直接开始写代码"~~ ❌
