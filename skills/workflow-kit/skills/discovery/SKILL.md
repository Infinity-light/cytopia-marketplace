## 核心定义

Discovery 不是线性的“调研 → 写 PRD”。

Discovery 的本质是：**对从当前状态到目标结果之间的全部可达路径，做充分勘测、拆解、比较、解释与对齐，并通过对话完成决策，持续消灭未决项，直到形成可进入 Planning 的无歧义共识。这里的“全部可达路径”必须覆盖常规路径、边缘路径、复杂路径、反常规路径，以及一切在当前现实约束下理论可达的路径；不得因为主观判断“太怪”“不像最佳实践”“大概率不会选”而提前过滤。只有逻辑上不可达、物理上不可达、约束上不可达的路径才可以排除，且排除时必须说明为什么不可达。**

PRD 只是最终固化结果，不是过程主体。Discovery 的主体发生在与用户的对话里。

## 第一原则

### 1. 必须做全路径勘测，不能只给单一路径

只给一个方案，不叫 Discovery，叫直接替用户拍板。

必须至少做到：
- 识别从当前状态到目标结果的全部可达路径
- 对全部可达路径进行全路径勘测，不得预先按“主要/次要”筛掉理论可达路径
- 展示每条可达路径的成立条件
- 说明每条路径为什么成立、为什么不同
- 明确哪些路径可立即推进，哪些路径受制于前置条件
- 对被排除的路径逐条说明其为何属于逻辑上不可达、物理上不可达或约束上不可达

如果客观上只有一条可行路径，也必须说明：
- 为什么只剩这一条
- 其他常见路径为什么不成立
- 这一判断依赖哪些前提

### 2. 路径差异必须落到变量上

路径差异不能停留在“方案 A 更快、方案 B 更稳”这种空泛表述，必须继续下钻到变量层。

变量至少包括：
- 时间成本
- 实现复杂度
- 维护成本
- 风险暴露面
- 性能/容量边界
- 扩展性
- 可逆性/切换成本
- 对现有系统/流程的侵入性
- 对用户体验的影响
- 对业务节奏的影响

每个变量都要继续解释两层：
1. 这个变量为什么会产生差异
2. 这个差异最终如何影响用户

### 3. 提问不是收集偏好，而是完成决策

AskUserQuestion 的目的不是“问问你喜欢哪个”，而是**帮助用户完成一个本来尚未完成的决策**。

所以每次提问必须包含：
- 当前待决事项是什么
- 可选项有哪些
- 每个选项的关键差异是什么
- 差异背后的变量是什么
- 这些变量会如何影响用户
- 建议用户基于什么标准做选择

禁止无信息量提问，例如：
- “你更喜欢哪个？”
- “要不要用 A？”
- “这个可以吗？”
- “你想快一点还是稳一点？”

必须把问题组织成可决策的问题，而不是可闲聊的问题。

### 4. Discovery 的核心任务是消灭未决项

Discovery 是否完成，不看写了多少字，只看**关键未决项是否清零**。

只要仍然存在会影响后续 Planning / Execution 的关键未决项，Discovery 就不能结束。

### 5. 只有在关键事项全部决定、无歧义后，才允许写文档

文档是已经达成共识后的固化物，不是拿来替代思考、替代对齐、替代决策的中间产物。

不允许：
- 一边还有关键分歧，一边提前写 PRD
- 用“先写个初稿再说”替代真正的 Discovery
- 把未决定事项原样丢进文档，假装完成 Discovery

## Discovery 的完成标准

Discovery 结束前，必须同时满足：

- [ ] 当前状态已被准确说明
- [ ] 目标结果已被准确说明
- [ ] 全部可达路径已完成一轮全路径勘测
- [ ] 每条可达路径的差异已落实到关键变量
- [ ] 关键变量对用户的影响已解释清楚
- [ ] 所有关键待决事项都已通过对话完成决策
- [ ] 不存在会影响后续规划与执行的关键歧义
- [ ] 用户已明确确认当前 Discovery 结论成立
- [ ] 此时才允许固化为文档

只要其中任一项不满足，Discovery 就未完成。

## 正确顺序

Discovery 必须严格遵循以下顺序，不得颠倒：

1. **达到可调研阈值**
2. **进行一轮全路径调研**
3. **向用户展示所有可达路径**
4. **解释路径差异、关键变量、变量对用户的影响**
5. **通过提问完成决策**
6. **持续清空未决项**
7. **形成无歧义共识**
8. **固化文档**
9. **输出文档路径**
10. **再问是否进入 Planning**

### 1）达到可调研阈值

在开始正式调研前，先确认已具备最基本的调研条件。至少要知道：
- 当前状态是什么
- 目标结果是什么
- 初始边界是什么
- 是否存在明显硬约束

如果这些基础信息都没有，就先通过对话补齐到“可调研阈值”，再开始调研。

### 2）进行一轮全路径调研

至少完成一轮覆盖全部可达路径的调研。这里的“全路径勘测”不是穷举逻辑上不可达、物理上不可达、约束上不可达的幻想路径，而是覆盖所有在当前现实约束下理论可达的路径，包括常规路径、边缘路径、复杂路径、反常规路径，以及看起来荒诞但只要理论可达就必须纳入的路径。不得因为主观判断“太怪”“不像最佳实践”“大概率不会选”而提前过滤。

调研输出必须包括：
- 路径清单
- 每条路径的成立条件
- 每条路径的关键变量
- 每条路径的主要收益与代价
- 每条路径对用户的具体影响
- 被排除路径及其不可达原因说明

### 3）展示所有可达路径

必须把全部可达路径完整展示给用户，而不是只展示推荐项。

推荐可以有，但推荐之前要先让用户看到：
- 有哪些全部可达路径
- 为什么是这些全部可达路径
- 它们哪里不同

### 4）解释差异 / 变量 / 用户影响

这一环节是 Discovery 的解释核心。

必须从“路径不同”继续拆到：
- 是哪些变量导致它们不同
- 每个变量如何改变实施方式、成本结构、风险结构、交付节奏
- 这些变化如何影响用户的目标达成、投入产出、后续自由度

### 5）通过提问完成决策

当路径差异已经解释清楚后，再用提问推动用户完成决策。

提问必须满足：
- 问的是决策，不是感受
- 问的是关键分叉，不是枝节偏好
- 问之前已经完成充分解释
- 问之后能够消灭至少一个关键未决项

### 6）持续清空未决项

在整个 Discovery 过程中，维护一个“未决项视图”。

每一轮对话后，都要判断：
- 哪些未决项已被解决
- 哪些未决项仍然存在
- 哪些新未决项被暴露出来
- 哪些未决项属于关键项，必须在结束前清零

关键未决项不清零，Discovery 不得收口。

### 7）形成无歧义共识

在写文档前，必须先做一次收口确认：
- 当前结论是否成立
- 是否准确反映双方已达成的决定
- 是否还存在影响后续阶段的关键歧义

**这一步必须先于写文档。**

### 8）固化文档

只有在无歧义共识形成后，才把 Discovery 结论写入 PRD。

### 9）输出文档路径

文档完成后，明确告知用户生成位置。

### 10）再问是否进入 Planning

在文档写完并给出路径后，再用 AskUserQuestion 询问是否进入 Planning。

## 提问规范

## 何时提问

只有在以下情况才提问：
- 需要用户对关键分叉做决定
- 需要用户补充无法通过调研获得的关键事实
- 需要用户确认当前 Discovery 结论已经成立
- 文档完成后，需要用户决定是否进入 Planning

如果问题可以通过进一步思考、读文件、搜索、调研解决，就先不要问用户。

## 如何提问

每次提问必须包含以下结构：

1. **待决事项**：当前到底在决定什么
2. **选项列表**：至少列出全部选项
3. **差异说明**：各选项关键差异是什么
4. **变量解释**：差异由哪些变量造成
5. **用户影响**：这些变量最终如何影响用户
6. **建议决策标准**：建议按什么标准选
7. **明确收口动作**：用户回答后，将消灭哪个未决项

### 合格示例

不合格：
- “数据库你想用 MySQL 还是 SQLite？”

合格：
- “当前待决事项是首版数据存储路径。全部可达路径中，至少包括以下全部选项：
  1. SQLite：部署最轻、启动最快，但并发能力和后续扩展空间有限；
  2. MySQL：前期配置更重，但更适合多人协作、长期增长和后续拆分。
  如果在当前现实约束下还存在其他理论可达路径，也必须一并纳入说明；只有逻辑不可达、物理不可达、约束不可达的路径才可以排除，并说明排除原因。以上差异的关键变量是部署成本、并发压力、后续扩展成本和迁移可逆性。对你而言，如果首版目标是尽快验证、单机运行、短期低并发，SQLite 会缩短交付时间；如果你预计很快进入多人使用或持续扩展，MySQL 会减少二次迁移成本。建议你按‘首版验证速度优先’还是‘中期扩展稳定性优先’来决定。你确认选择哪条路径？确认后，这个关键未决项就会清零。”

## 输出物：PRD 仅作为最终固化

## 文档命名规范

PRD 文件必须使用时间戳命名，格式如下：

```text
[项目名]-[YYYYMMDD-HHmm]-PRD.md
```

存储位置固定为：

```text
.claude/discovery/
```

完整路径格式：

```text
.claude/discovery/[项目名]-[YYYYMMDD-HHmm]-PRD.md
```

示例：
- `.claude/discovery/舆情监测系统-20260308-1430-PRD.md`
- `.claude/discovery/支付网关改造-20260308-2215-PRD.md`

禁止继续使用无时间戳、无项目名、语义含混的命名方式。

## PRD 必须记录的内容

PRD 不是过程记录，而是**已决结论的固化**。文档中只能写：
- 已确认的目标
- 已确认的边界
- 已比较过的全部可达路径及结论
- 关键决策与理由
- 已清零的关键未决项结果
- 后续 Planning 所需的无歧义输入

PRD 中不应保留“待定 / 再看 / 后面再说”这种关键悬而未决项。

### 推荐结构

```markdown
# PRD: [项目名]

## 1. 当前状态
- [当前现状]

## 2. 目标结果
- [目标结果]

## 3. 约束与边界
- 包含：[内容]
- 不包含：[内容]
- 硬约束：[内容]

## 4. 全部可达路径对比

### 路径 A：[
路径名称]
- 成立条件：[内容]
- 关键变量：[内容]
- 对用户的影响：[内容]
- 结论：[内容]

### 路径 B：[
路径名称]
- 成立条件：[内容]
- 关键变量：[内容]
- 对用户的影响：[内容]
- 结论：[内容]

## 5. 关键决策
- 决策 1：[结论]
  - 决策理由：[内容]
  - 对用户的影响：[内容]
- 决策 2：...

## 6. 已清零的关键未决项
- [未决项] → [决议结果]

## 7. 进入 Planning 的前置共识
- [供 Planning 直接使用的无歧义结论]
```

如无必要，不要为了“看起来完整”而堆砌空章节。

## 行为约束

### 必做

- 必须覆盖全部可达路径，并完成至少一轮全路径勘测
- 必须覆盖常规、边缘、复杂、反常规、看起来荒诞但在当前现实约束下理论可达的路径
- 不允许因为主观判断“太怪”“不像最佳实践”“大概率不会选”而提前过滤任何理论可达路径
- 只有逻辑不可达、物理不可达、约束不可达的路径才可以排除，且排除时必须说明原因
- 必须解释路径差异背后的关键变量
- 必须解释变量如何影响用户
- 必须把提问组织成决策问题
- 必须持续识别并清空关键未决项
- 必须先确认 Discovery 结论成立，再写文档
- 必须在文档写完后输出文档路径
- 必须在最后再问是否进入 Planning

### 禁止

- 禁止只给单一路径就要求用户选
- 禁止把 PRD 当作 Discovery 主体
- 禁止关键事项未决就提前写文档
- 禁止提出没有差异解释的裸问题
- 禁止把“用户没反对”当作“用户已决定”
- 禁止在存在关键歧义时结束 Discovery
- 禁止跳过 Planning 直接进入 Execution

## 阶段结束动作

### 第一步：收口确认当前 Discovery 结论是否成立

在写文档前，必须先通过 AskUserQuestion 做一次结论确认。

第一次 AskUserQuestion（写 PRD 之前）选项必须硬编码为：
- “当前 Discovery 结论成立（推荐）”
- “需要补充调研”
- “需要修正理解”

确认重点：
- 当前状态描述是否成立
- 目标结果描述是否成立
- 全部可达路径与差异理解是否成立
- 关键决策是否已经完成
- 是否仍有关键未决项

只有当用户明确选择“当前 Discovery 结论成立（推荐）”后，才允许写 PRD。
如果用户选择“需要补充调研”或“需要修正理解”，必须继续 Discovery，不得写 PRD。
禁止用“是否进入 Planning”来替代这一次结论确认。

### 第二步：写入 PRD

当且仅当上述确认完成后，将结论固化到：

```text
.claude/discovery/[项目名]-[YYYYMMDD-HHmm]-PRD.md
```

### 第三步：输出文档路径

明确把 PRD 路径返回给用户。

### 第四步：确认是否进入 Planning

文档写完后，再使用 AskUserQuestion 询问下一步。

第二次 AskUserQuestion（PRD 写完后）选项必须硬编码为：
- “进入 Planning（推荐）”
- “继续补充 Discovery”
- “暂停”

不允许的选项：
- “跳过 Planning 直接进入 Execution”
- “直接开始写代码”

Discovery 的职责到此结束。
