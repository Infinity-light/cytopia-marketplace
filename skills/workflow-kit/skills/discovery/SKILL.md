## 核心方法论：三部曲

Discovery 严格遵循"为什么 → 是什么 → 怎么办"：

| 阶段 | 核心问题 | 产出 |
|------|---------|------|
| **为什么** | 用户到底想要什么？ | 清晰的需求边界、交付物清单、验收标准 |
| **是什么** | 这个东西本质上怎么回事？ | **最小模块拆解** + **三重维度尽调** |
| **怎么办** | 具体如何组织实现？ | **用户动线梳理** + **细粒度步骤** |

---

## 第一部曲：为什么（Why）

**目标**：透彻探明用户需求，具体到交付物的每一个页面和最小呈现节点。

### 动作清单

1. **接收用户表达**
   - 不打断、不急于给方案
   - 完整听完用户说什么

2. **概念澄清**
   - 对关键概念、术语、引用逐一查证
   - 不确定就先查（读文件、搜索、问用户）
   - 不假设、不替代、不猜测

3. **需求拆解**
   - 目标：用户想达成什么
   - 动机：为什么要做
   - 约束：限制条件是什么
   - 边界：哪些内容不在范围内

4. **交付物具象化**
   - 若为产品/系统：拆出每个页面、模块
   - 若为功能：描述每个交互节点输入输出
   - 若为服务：明确调用方与使用场景
   - **拆解到最小可验证单元**（单个子代理可独立完成）

### 检查点

通过以下检查才能进入第二部曲：

- [ ] 用户目标已清晰
- [ ] 约束条件已确认
- [ ] 边界范围已划定
- [ ] 交付物已具象到最小节点
- [ ] 没有“我不确定用户想要什么”的残留疑问

## 第二部曲：是什么（What）

**核心重点**：
1. **最小模块拆解**：将需求拆分到不可再分的原子单元
2. **三重维度尽调**：对每个最小模块做深度、广度、前瞻性调研
3. **最前沿方案**：找到当前阶段可落地且先进的实现路径

### 第一步：最小模块拆解

将需求拆解为树状结构，直到不可再分：

```
用户需求
├── 模块 A
│   ├── A1: 最小功能单元
│   ├── A2: 最小功能单元
│   └── A3: 最小功能单元
├── 模块 B
│   ├── B1: 最小功能单元
│   └── B2: 最小功能单元
└── 模块 C
    ├── C1: 最小功能单元
    └── C2: 最小功能单元
```

**拆解标准**：
- 每个最小单元可由单个子代理独立完成
- 每个最小单元有明确输入输出
- 每个最小单元对应可比较的技术方案

### 第二步：三重维度尽调

对每个最小模块并行尽调：

| 维度 | 调研内容 | 产出要求 | 工具 |
|------|---------|---------|------|
| **深度** | 技术原理、架构设计、实现细节、性能特征 | 原理解释 + 代码示例 | context7 + 官方文档 |
| **广度** | 主流方案对比（开源/商业/云原生）、适用场景、成本分析 | 对比表（≥3种） | WebSearch + GitHub |
| **前瞻性** | 技术演进趋势、新兴方案、潜在主流方向 | 趋势判断 + 推荐理由 | 论文/社区资料 |

**尽调深度要求**：
- 开源项目优先看活跃度（stars、最近提交）
- 每个方案明确优缺点、适用场景、学习与维护成本
- 通过 issues/讨论区识别常见坑点
- 关注 2024-2026 年关键技术变化

**产出**：每个最小模块需形成尽调结论（深度 + 广度 + 前瞻性）。

## 第三部曲：怎么办（How）

**核心重点**：
1. **用户动线梳理**：从起点到终点的完整流程
2. **细粒度步骤**：每一步可执行
3. **逻辑结构组织**：架构和数据流清晰

### 第一步：梳理用户动线

```
用户起点（输入/触发）
    ↓
步骤 1: [具体操作]
    ↓
步骤 2: [具体操作]
    ↓
步骤 3: [具体操作]
    ↓
用户终点（输出/结果）
```

**动线要求**：
- 从用户第一个动作开始
- 每步具体到可执行级别
- 明确决策分支（若 X 则 A，否则 B）
- 上一步输出可作为下一步输入

### 第二步：逻辑结构设计

```
┌─────────────────────────────────────────┐
│              系统架构图                  │
├─────────────────────────────────────────┤
│  输入层 → 处理层 → 存储层 → 输出层       │
│     ↓        ↓        ↓        ↓        │
│   [细节]   [细节]   [细节]   [细节]     │
└─────────────────────────────────────────┘
```

**架构要求**：
- 数据流清晰
- 模块边界清晰
- 接口输入输出明确
- 异常处理路径可见

### 第三步：呈现与确认

1. **展示调研结论**
   - 每个最小模块的尽调结论
   - 方案对比与推荐理由

2. **展示动线与架构**
   - 用户动线
   - 系统架构与数据流

3. **用户确认（AskUserQuestion）**
   - 关键取舍必须让用户明确选择
   - 不接受“都行”

### 前端范围约束（强制）

Discovery 中涉及前端时，**仅做探索性工作**：
- 样式方向（style）
- 展现结构与信息架构（presentation / IA）
- 交互路径与状态设计（interaction）

**禁止在 Discovery 阶段进行真实前端开发行为**：
- 不写前端业务代码
- 不创建前端工程文件
- 不落地页面实现

### 检查点

- [ ] 每个最小模块的三重尽调已完成
- [ ] 用户动线已梳理
- [ ] 系统架构已设计
- [ ] 前端范围已保持在探索层（如适用）
- [ ] 用户已做出明确选择

## 输出：PRD 文档

### 命名规范（强制）

PRD 文件必须使用时间戳命名，格式如下：

```
[项目名]-[YYYYMMDD-HHmm]-PRD.md
```

示例：
- `舆情监测系统-20260307-1430-PRD.md`
- `Claude安装器打包-20260307-0915-PRD.md`

**存储位置（强制）**：`.claude/discovery/[项目名]-[YYYYMMDD-HHmm]-PRD.md`

---

将三部曲成果写入该 PRD 文件，结构至少包含：

```markdown
# PRD: [项目名称]

## 一、需求理解（为什么）
- 目标
- 交付物清单（最小单元）
- 边界条件

## 二、调研结论（是什么）
- 各最小模块的深度/广度/前瞻性结论
- 方案对比与推荐理由

## 三、实现路径（怎么办）
- 用户动线
- 系统架构
- 数据流与关键接口

## 四、验收标准
- 验收项清单
```

---

## 阶段结束

用 AskUserQuestion 向用户确认下一步，选项必须是：
- **进入 Planning（推荐）**
- **需要补充调研**

### Planning 交接要求（强制）

Discovery 结束时必须明确输出 PRD 交接路径，格式建议：

```
PRD 文件路径：.claude/discovery/[项目名]-[YYYYMMDD-HHmm]-PRD.md
```

未给出明确 PRD 路径，不得视为可进入 Planning。
